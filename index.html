Post-mortem constructivo: error 403 en la integración de API de mapas de Eco-Track

Contexto

Eco-Track es un proyecto orientado al seguimiento de rutas y ubicaciones mediante una API de mapas. Durante el cierre de un sprint, el equipo estaba trabajando en el ticket ECO-42: Integración de API de mapas.

El objetivo de la tarea era conectar el frontend con el servicio de mapas para mostrar ubicaciones dentro de la aplicación. La integración era importante porque permitía avanzar con pruebas de QA y validar una funcionalidad central del producto.

El equipo trabajaba de forma remota, utilizando Slack para comunicación asincrónica, Jira para seguimiento de tickets y GitHub para control de versiones.

Problema

Durante la integración, el auth_token utilizado para consumir la API de mapas comenzó a devolver un error 403.

El error 403 indica que la solicitud fue recibida por el servidor, pero no tiene permisos suficientes para acceder al recurso solicitado.

El problema bloqueó el avance del ticket porque no se podía confirmar si la causa era:

  • token vencido;
  • permisos insuficientes;
  • cambio en el endpoint;
  • documentación interna desactualizada;
  • configuración incorrecta del ambiente de staging.

Desde una perspectiva no técnica, el impacto era claro: la funcionalidad de mapas no podía probarse ni validarse, por lo tanto, QA no podía avanzar con la revisión del flujo completo.

Acciones tomadas

Para abordar el incidente, se aplicó un enfoque de investigación ordenado y sin culpas.

Primero, se documentó el bloqueo en el ticket ECO-42, incluyendo el contexto, el error observado y los intentos realizados.

  • Verificación del header donde se enviaba el token.
  • Prueba del endpoint desde Postman.
  • Revisión de la documentación interna.
  • Comparación de variables del archivo .env.
  • Regeneración del token en ambiente de staging.
  • Consulta asincrónica al equipo de Backend/Auth.
  • Registro del bloqueo en Jira.
  • Comunicación del impacto a QA y Tech Lead.
  • Propuesta de actualizar la documentación técnica.

Mensaje asincrónico utilizado

Estoy trabajando en el ticket ECO-42 de integración de API de mapas. El auth_token devuelve error 403 y la documentación interna parece estar desactualizada. Ya probé regenerar el token, validar headers y revisar el endpoint desde Postman. ¿Alguien de Backend/Auth podría confirmar si el token actual tiene permisos vigentes o si cambió el flujo de autenticación? Necesito confirmarlo antes de las 16:00 para no afectar las pruebas de QA.

Post-mortem constructivo

Qué ocurrió

El equipo detectó que la documentación interna sobre autenticación de la API de mapas no estaba actualizada. Esto generó confusión sobre el uso correcto del token y los permisos necesarios.

Causa raíz probable

La causa principal fue una combinación de documentación desactualizada y falta de una validación clara sobre permisos del token en ambiente de staging.

Factores contribuyentes

  • No existía una sección específica sobre errores frecuentes de autenticación.
  • El flujo para regenerar o validar tokens no estaba documentado.
  • El ticket no indicaba a quién escalar problemas de permisos.
  • No había un checklist previo para validar integraciones externas.

Qué funcionó bien

  • El bloqueo se comunicó de forma temprana.
  • Se documentaron los intentos realizados antes de pedir ayuda.
  • El equipo evitó convocar una reunión innecesaria al inicio.
  • Se dejó trazabilidad en Jira y Slack.
  • El feedback se mantuvo enfocado en el proceso y no en personas.

Qué se puede mejorar

  • Actualizar documentación técnica después de cambios en autenticación.
  • Agregar una sección de errores frecuentes.
  • Definir responsables para validar permisos de APIs externas.
  • Crear un checklist previo para integraciones.
  • Establecer un proceso de revisión de documentación antes del cierre de sprint.

Decisiones tomadas

Acción Responsable Deadline
Validar permisos del token en staging Backend/Auth Antes del cierre del sprint
Actualizar documentación de autenticación Tech Lead / Backend Próximo sprint
Agregar sección de errores frecuentes Equipo técnico Próximo sprint
Crear checklist para integraciones externas Frontend + Backend Próxima retrospectiva
Documentar el caso como aprendizaje Persona responsable del ticket Al cierre del PR

Aprendizajes

Este incidente permitió identificar que no todos los bloqueos técnicos se resuelven únicamente revisando código. A veces, el problema está en la falta de documentación clara, en procesos incompletos o en dependencias entre equipos.

  • Documentar bien ahorra tiempo.
  • Un mensaje claro reduce reuniones innecesarias.
  • La evidencia visual y los pasos reproducibles aceleran la resolución.
  • Los post-mortems deben buscar aprendizaje, no culpables.
  • El control de versiones permite dejar trazabilidad del proceso.
  • La comunicación asincrónica funciona mejor cuando hay contexto, acción y deadline.
  • La seguridad importa: no se deben compartir tokens ni credenciales en canales abiertos.

Uso de control de versiones

Para documentar el proceso, se utilizó GitHub como herramienta de control de versiones. Se creó una rama específica llamada feature/postmortem-api-maps-403 y se realizaron commits descriptivos para evidenciar la evolución del blog técnico.

Reflexión sobre feedback radicalmente sincero

Durante el proceso, el feedback se aplicó siguiendo los principios de Radical Candor, combinando cuidado personal con desafío directo.

El desafío directo consistió en señalar que la documentación técnica estaba incompleta y que eso estaba afectando el avance del ticket. Sin embargo, el mensaje se formuló sin culpar a una persona específica. Se habló del proceso, del impacto y de las mejoras necesarias.

Un ejemplo de feedback aplicado sería:

Gracias por la documentación inicial de la API de mapas. Me ayudó a entender el flujo general, pero durante la integración detecté que falta información sobre autenticación, permisos del token y manejo del error 403. Esto generó un bloqueo en el ticket ECO-42. ¿Podríamos sumar una sección con esos casos para que futuras integraciones sean más claras?

Este enfoque permite comunicar un problema real sin atacar a nadie. Además, transforma el error en una oportunidad de aprendizaje compartido.

Conclusión

El error 403 en la integración de la API de mapas fue un bloqueo técnico, pero también una oportunidad para mejorar la documentación, la comunicación asincrónica y los procesos de escalamiento.

El post-mortem constructivo permitió identificar causas, aprendizajes y acciones preventivas. Además, el uso de GitHub aportó trazabilidad al proceso y ayudó a mostrar cómo evolucionó la documentación.

En equipos remotos, comunicar con claridad, documentar decisiones y dar feedback sincero con empatía son habilidades tan importantes como resolver el problema técnico en sí.
